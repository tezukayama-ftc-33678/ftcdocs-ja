# フェーズ細分化ガイド

## 📋 目的

大規模な翻訳フェーズを管理可能なサブフェーズに細分化するためのガイドラインです。

## 🎯 細分化の基準

### 1. 作業量による分割
- **1サブフェーズの目安**: 15-25時間（約1-2週間の作業、1日2-3時間想定）
- **ファイル行数の目安**: 1,500-2,500行
- **ファイル数の目安**: 5-10ファイル

### 2. 論理的なまとまりによる分割
- 機能やトピックごとにグループ化
- 依存関係のあるファイルは同じサブフェーズに含める
- ユーザーが順番に読むであろう順序を考慮

### 3. 優先順位による分割
- 最も使用頻度の高いドキュメントを先に
- 基本的な概念説明を先に、詳細な手順を後に
- メインページ → 詳細チュートリアル → 補足資料の順

## 📝 Phase 4 細分化の例

Phase 4（プログラミングリソース）は、以下のように4つのサブフェーズに分割されました：

### Phase 4a: メインページと SDK ドキュメント
- **対象**: 19ファイル
- **工数**: 約5時間
- **内容**: 各プログラミング環境のメインページ、SDK 概要と更新手順
- **理由**: ユーザーが最初に読むページ、全体像の理解に必要

### Phase 4b: Blocks プログラミング詳細チュートリアル
- **対象**: 6ファイル、約1,500行
- **工数**: 15-20時間
- **内容**: Blocks 環境の詳細な手順説明
- **理由**: 最も初心者向けで使用頻度が高い

### Phase 4c: OnBot Java 詳細チュートリアル
- **対象**: 4ファイル、約1,400行
- **工数**: 15-20時間
- **内容**: OnBot Java 環境の詳細な手順説明
- **理由**: 中級者向け、Blocks の次のステップ

### Phase 4d: Android Studio 詳細チュートリアル
- **対象**: 8ファイル、約2,000行
- **工数**: 20-25時間
- **内容**: Android Studio 環境の詳細な手順説明
- **理由**: 上級者向け、最も複雑な環境

## 🔄 他のフェーズへの適用手順

### ステップ1: フェーズの分析
```bash
# ファイルリストと行数を取得
find docs/source/[target_directory] -name "*.rst" -exec wc -l {} + | sort -n
```

### ステップ2: 論理的なグループ化
- 関連するファイルをグループ化
- 各グループの行数を合計
- 15-25時間の作業量になるように調整

### ステップ3: サブフェーズの定義
各サブフェーズに以下を定義：
- サブフェーズ番号（例: 5a, 5b, 5c）
- 目標（何を翻訳するか）
- 対象ファイルリスト
- 予想工数
- 優先順位

### ステップ4: TRANSLATION_ROADMAP.md の更新
```markdown
### **Phase X: [フェーズ名]** (優先度: [高/中/低])
**目標:** [フェーズの全体目標]

**注**: Phase X は作業量が大きいため、Nつのサブフェーズに分割して実施

---

#### **Phase Xa: [サブフェーズ名]** ✅/⏳ - **完了/進行中/未着手**
**目標:** [サブフェーズの目標]

##### 翻訳対象ファイル（Nファイル、約X,XXX行）
- [ ] ファイル1 (XXX行)
- [ ] ファイル2 (XXX行)

##### 予想工数: XX-XX時間

---
[他のサブフェーズも同様に...]
```

### ステップ5: サブフェーズ完了サマリーの作成
各サブフェーズ完了時に `PHASEXa_SUMMARY.md` を作成：
- 完了した作業の詳細
- 翻訳統計
- 次のサブフェーズへのポインター

## 📊 推奨される細分化パターン

### パターン1: 環境別分割
複数のプログラミング環境がある場合：
- メインページをまとめて1つのサブフェーズ
- 各環境の詳細を別々のサブフェーズに

**例**: Phase 4（Blocks、OnBot Java、Android Studio）

### パターン2: トピック別分割
同じ環境内の異なるトピック：
- 基本概念
- 設定とインストール
- 実装チュートリアル
- 高度な機能

**例**: Phase 5（AprilTag、カラー処理、ビジョンプログラミング）を想定

### パターン3: 難易度別分割
初心者向けから上級者向けへ：
- 入門と基本
- 中級機能
- 高度な使用法

### パターン4: ハイブリッド分割
上記の組み合わせ：
- メインページ + 基本ドキュメント
- 環境A詳細
- 環境B詳細
- 高度な機能全般

## ⚠️ 注意事項

### 細分化しすぎない
- サブフェーズが多すぎると管理が煩雑
- 1フェーズは2-5サブフェーズ程度が適切

### 依存関係に注意
- 前のサブフェーズの内容を参照するファイルは後のサブフェーズに

### PR の粒度
- 1つのサブフェーズ = 1つのPR
- レビュー可能な規模に保つ

### ブランチ戦略
各サブフェーズに対して：
```bash
git checkout main
git pull origin main
git checkout -b copilot/execute-phase-Xa-[description]
# 作業...
git push origin copilot/execute-phase-Xa-[description]
# PR作成
```

## 📈 進捗管理

### マイルストーン設定
- 各サブフェーズをGitHub マイルストーンとして設定
- 完了したサブフェーズにチェックマーク

### 状態表記
- ✅ 完了
- 🔄 進行中
- ⏳ 未着手
- 🔴 ブロック（依存関係待ち）

## 🎓 ベストプラクティス

1. **早期の細分化**: フェーズ開始前に細分化計画を立てる
2. **柔軟な調整**: 作業中に必要に応じてサブフェーズを調整
3. **明確なマイルストーン**: 各サブフェーズの完了基準を明確に
4. **継続的なレビュー**: 各サブフェーズ完了後にレビューを実施
5. **ドキュメント更新**: TRANSLATION_ROADMAP.md を常に最新に保つ
